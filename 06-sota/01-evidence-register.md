# Evidence Register: claims, данные, SoTA-статус

**Назначение:** Трассировка "claim -> данные -> метод расчёта -> SoTA-статус -> критерий пересмотра".  
**Читай это, если:** Нужно понять, почему claim считается рабочим и когда его статус должен измениться.

---

## 1) Контур применимости

- **Срез:** пилотные семьи 3–10 лет (дошкольный и школьный треки MVP).
- **Период:** обновляется волнами (указать даты в строках claim).
- **Минимальные источники:** weekly logs, mastery map по завершению цикла, adult reflection.

## 4) Правила обновления

1. Любой новый числовой claim получает `Claim ID`.
2. Без данных и формулы claim не поднимается в публичные документы.
3. У каждого claim обязателен `SoTA статус` и `Revision criterion`.
4. При изменении формулы/окна/среза обновляется строка claim и дата ревизии.
5. Для публичного слоя публикуем только claim с явным уровнем надёжности (F-G-R + SoTA).

---

## 5) Разделение осей (важно)

- **SoTA статус** = актуальность утверждения (`current/preliminary/hypothesis/outdated`).
- **Maturity статуса внедрения** = стадия проекта (`MVP/pilot/scaled`).

Эти оси не объединяются в одну метку.
